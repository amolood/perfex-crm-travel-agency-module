# موديول وكالة السفر (Travel Agency)

**الإصدار:** 1.0.0
**يتطلب Perfex CRM:** 2.3.* فأعلى

موديول متكامل لإدارة وكالات السفر داخل Perfex CRM: باقات سياحية، حجوزات عملاء، تفويج (مجموعات سفر) مع خط سير وترانسبورت وجوازات وتأشيرات، موردين، وحسابات/سدادات الموردين — بالإضافة إلى بوابة عميل لعرض الحجوزات والتذاكر.

---

## 1. الفكرة العامة

الموديول يغطي دورة عمل وكالة السفر بالكامل:

1. تعريف **أنواع الرحلات** (سياحة، حج، عمرة... إلخ) و**الموردين** (فنادق، شركات نقل، وكلاء تذاكر...).
2. إنشاء **باقات سياحية** مرتبطة بمورد ونوع رحلة، تحمل سعر البيع للعميل وتكلفة الشراء من المورد.
3. تسجيل **حجوزات العملاء** على هذه الباقات، مع إمكانية إنشاء فاتورة تلقائياً في نظام الفواتير بنفس عملية الحجز.
4. تجميع الحجوزات ضمن **مجموعات سفر (تفويج)** لإدارة خط السير (المدن/الفنادق)، وسائل النقل، وبيانات المسافرين (الجواز، التأشيرة، الصور).
5. متابعة **حسابات الموردين**: كم مستحق لكل مورد (بناءً على تكلفة الباقات المحجوزة)، كم تم سداده فعلياً، والرصيد المتبقي — لكل عملة على حدة.
6. عرض هذه البيانات للعميل نفسه عبر **بوابة العميل** (My Bookings).

عنصر رئيسي باسم **Travel Agency** (أيقونة طائرة)، يظهر فقط لمن يملك صلاحية `view` على قدرة `travel_agency`، ويحتوي العناصر الفرعية التالية بالترتيب:

| العنصر | الرابط | ملاحظة |
|---|---|---|
| الباقات (Packages) | `travel_agency/packages` | |
| أنواع الباقات (Package Types) | `travel_agency/package_types` | |
| الحجوزات (Bookings) | `travel_agency/bookings` | |
| المجموعات / التفويج (Groups) | `travel_agency/groups` | |
| الموردون (Suppliers) | `travel_agency/suppliers` | يتطلب صلاحية منفصلة `travel_agency_suppliers` |

الموديول يسجّل مجموعتي صلاحيات مستقلتين لكل من `view`/`create`/`edit`/`delete`:

- **`travel_agency`** — تحكم في الباقات، أنواع الباقات، الحجوزات، والمجموعات (التفويج).
- **`travel_agency_suppliers`** — تحكم منفصل في الموردين وحساباتهم وسداداتهم (بحيث يمكن منح موظف صلاحية الحجوزات دون رؤية بيانات الموردين المالية والعكس).

## 3. أنواع الباقات (Package Types)

تصنيف بسيط لغرض الرحلة، قابل للتفعيل/التعطيل وله ترتيب عرض (`display_order`). عند تثبيت الموديول يتم زرع أنواع افتراضية جاهزة:

- السياحة
- الحج
- العمرة
- إقامة عمل
- زيارة عائلية
- زيارة شخصية

لا يمكن حذف نوع مستخدم فعلياً في أي باقة (يُرفض الحذف تلقائياً لمنع تكسّر البيانات).

---

## 4. الباقات السياحية (Packages)

كل باقة تحتوي:

- الاسم، الوجهة (destination)، الوصف
- **نوع الرحلة** (من أنواع الباقات) و**المورد** المرتبط بها
- **مدة الرحلة** بالأيام، **تاريخ البداية** و**تاريخ النهاية**
- **عدد المقاعد المتاحة**
- **العملة** (تُحدَّد لكل باقة، وتُستخدم لاحقاً في الفاتورة وحساب المورد)
- **سعر البيع للعميل (price)** و**تكلفة الشراء من المورد (cost)** — منفصلان تماماً
- حالة التفعيل (نشطة/غير نشطة)، وتُستخدم الباقات النشطة فقط في قوائم الاختيار عند الحجز

### ضبط سعة المقاعد

قيمة **"عدد المقاعد المتاحة" = 0** تعني مقاعد **غير محدودة** لهذه الباقة. أي قيمة أكبر من صفر تُفعّل تحققاً صارماً من السعة: عند إضافة أو تعديل أي حجز (من لوحة التحكم أو من بوابة العميل)، يتحقق النظام أن (عدد المسافرين في الحجوزات الحالية + عدد مسافري الحجز الجديد) لا يتجاوز السعة المحددة، وإلا يُرفض الحجز برسالة "لا تتوفر مقاعد كافية" بدلاً من إتمامه. نفس التحقق يُطبَّق عند إضافة عضو جديد لمجموعة (تفويج).

### حساب الربح لكل مقعد

يوفر الموديول حساباً تلقائياً للربحية لكل باقة:

- **الربح لكل مقعد** = سعر البيع − التكلفة
- **هامش الربح %** = ((سعر البيع − التكلفة) ÷ سعر البيع) × 100

يظهر هذا في قائمة الباقات كمؤشر سريع لربحية كل باقة دون الحاجة لحسابات يدوية.

لا يمكن حذف باقة مرتبط بها حجوزات فعلية (يُرفض الحذف لحماية سجل الحجوزات والفواتير).

## 5. الحجوزات (Bookings)

نموذج الحجز يسجّل: العميل، الباقة، عدد المسافرين، تاريخ السفر، الحالة، وملاحظات.

### اختيار عميل موجود أو إضافة عميل جديد مباشرة

عند إنشاء حجز جديد، يوفر النموذج خيارين (Toggle):

1. **عميل مسجَّل مسبقاً** — يُختار من قائمة العملاء الحاليين مباشرة.
2. **إضافة عميل جديد بسرعة** — تظهر حقول مختصرة (اسم الشركة، البريد الإلكتروني، الاسم الأول، الاسم الأخير، رقم الهاتف) فيتم إنشاء العميل تلقائياً في نظام العملاء (Clients) أثناء حفظ الحجز نفسه، بدون الخروج من صفحة الحجز، ثم يُربط الحجز بالعميل الجديد مباشرة. يتطلب هذا الخيار أن يملك الموظف صلاحية `create` على وحدة العملاء (`customers`).

### حساب الإجمالي والفوترة التلقائية

- **إجمالي الحجز (total)** = سعر بيع الباقة (price) × عدد المسافرين — يُحسب تلقائياً ولا يُدخل يدوياً.
- عند تفعيل خيار **"إنشاء فاتورة"** أثناء الحجز، يقوم النظام تلقائياً بـ:
  - إنشاء فاتورة جديدة في نظام الفواتير لنفس العميل، بعملة الباقة (أو العملة الأساسية للنظام إن لم تُحدَّد عملة للباقة)
  - بند واحد بوصف "اسم الباقة - الوجهة"، بكمية = عدد المسافرين، وسعر الوحدة = سعر بيع الباقة
  - إجمالي الفاتورة = سعر البيع × المسافرين (**وليس** تكلفة المورد أبداً — الفاتورة دائماً على أساس سعر البيع)
  - ربط رقم الفاتورة الناتج بسجل الحجز تلقائياً (`invoiceid`)

### حالات الحجز

| الحالة | الوصف |
|---|---|
| قيد الانتظار (Pending) | الحالة الافتراضية عند الإنشاء |
| مؤكد (Confirmed) | |
| ملغي (Cancelled) | **يُستثنى تلقائياً من حسابات مستحقات الموردين** (انظر قسم حسابات الموردين) |
| مكتمل (Completed) | |

يمكن للموظف تحديث حالة الحجز مباشرة دون فتح نموذج التعديل الكامل.

## 6. المجموعات / التفويج (Groups)

هذه هي الوحدة التشغيلية لتنظيم رحلات جماعية على باقة معينة، وتشمل:

### بيانات المجموعة الأساسية
الاسم، الباقة المرتبطة، تاريخ المغادرة، تاريخ العودة، إجمالي المقاعد، ملاحظات، والحالة.

### حالات المجموعة

| الحالة |
|---|
| تخطيط (Planning) — الافتراضية |
| مؤكدة (Confirmed) |
| غادرت (Departed) |
| مكتملة (Completed) |
| ملغاة (Cancelled) |

### خط السير (Itinerary) — متعدد المحطات

بدلاً من حقل فندق/مدينة واحد ثابت، تدعم كل مجموعة عدد غير محدود من "محطات خط السير": **مدينة، عدد أيام الإقامة، اسم الفندق**، بترتيب عرض قابل للتعديل (يمكن إضافة/حذف محطات بحرية — سفرة بمحطة واحدة أو بعدة مدن).

### النقل (Transport) — متعدد الناقلين

بنفس المنطق، تدعم كل مجموعة عدة "أرجل نقل": **اسم الناقل، نوع النقل (طيران/بري/بحري...)، رقم مرجعي للحجز/الرحلة**، بدلاً من ناقل واحد ثابت — مفيد للرحلات متعددة القطاعات (مثلاً ذهاب بخط، عودة بخط آخر، أو نقل داخلي بري بين المدن).

### أعضاء المجموعة (المسافرون)

يتم إضافة المسافرين للمجموعة عبر **اختيار من الحجوزات المؤهلة** (حجوزات قائمة على نفس باقة المجموعة ولم تُضَف بعد كعضو فيها) — بذلك يبقى كل مسافر مربوطاً بحجزه وفاتورته الأصلية. تخضع إضافة الأعضاء لنفس تحقق سعة المقاعد المذكور في قسم الباقات.

لكل عضو/مسافر تُحفظ:

- الاسم الكامل
- رقم جواز السفر وتاريخ انتهائه
- **حالة التأشيرة**: لم تُقدَّم / قُدِّمت / معتمدة / مرفوضة
- رقم التأشيرة
- بيانات إضافية مستخرجة من الجواز (MRZ): الاسم الأخير، الأسماء الأولى، الجنسية، تاريخ الميلاد، الجنس، والنص الخام لشريط MRZ
- **صورة شخصية** و**صورة/مسح ضوئي لجواز السفر** — تُرفع وتُخزَّن في مجلد خاص لكل عضو، ويتم الوصول إليها فقط عبر رابط تحميل محمي بالصلاحيات (وليس رابطاً عاماً مباشراً)
- ملاحظات

يمكن تحديث حالة التأشيرة لكل مسافر بشكل مستقل وسريع دون فتح كامل نموذج التعديل.

> **ملاحظة مهمة:** بيانات التفويج (خط السير، النقل، الجوازات، التأشيرات) هي بيانات **داخلية للموظفين فقط** ولا تظهر للعميل إطلاقاً في بوابة العميل.

## 7. الموردون (Suppliers)

سجل بسيط لكل مورد: الاسم، النوع، البريد الإلكتروني، الهاتف، المدينة، العنوان، ملاحظات، وحالة التفعيل.

لا يمكن حذف مورد مرتبطة به باقات فعلية.

## 8. حسابات الموردين وسداداتهم (Supplier Accounts & Payments)

هذه من أهم ميزات الموديول: متابعة مالية دقيقة لما هو **مستحق للمورد** مقابل ما **تم سداده فعلياً له**.

### صفحة نظرة عامة مخصصة وسهلة الوصول

صفحة **"حسابات الموردين"** (Supplier Accounts) في القائمة الجانبية مباشرة (وليست مدفونة داخل شاشة كل مورد) تعرض **كل الموردين دفعة واحدة**، لكل مورد وكل عملة يتعامل بها:

- **المستحق (Due)**
- **المدفوع (Paid)**
- **الرصيد (Balance)**

مع تلوين واضح: الرصيد الموجب (مبلغ لا يزال مستحقاً للمورد) يظهر باللون الأحمر، والرصيد صفر أو أقل يظهر باللون الأخضر — بحيث يمكن اكتشاف الموردين الذين يستحقون سداداً بنظرة واحدة دون الدخول لكل مورد على حدة.

### آلية الحساب

- **المستحق (Due)** = مجموع (تكلفة الباقة (cost) × عدد المسافرين) لكل حجوزات ذلك المورد، **باستثناء الحجوزات الملغاة**، مجمّعة حسب العملة.
- **المدفوع (Paid)** = مجموع كل سدادات المورد المسجَّلة، مجمّعة حسب العملة.
- **الرصيد (Balance)** = المستحق − المدفوع.

ملاحظة جوهرية: المستحق للمورد يُحسب دائماً من **تكلفة الشراء (cost)**، بينما فاتورة العميل تُحسب من **سعر البيع (price)** — الفرق بينهما هو هامش ربح الوكالة، ولا يختلط الحسابان أبداً.

### تسجيل السدادات

من داخل صفحة كل مورد، قسم مخصص "السدادات" يتيح:

- عرض جميع السدادات السابقة لهذا المورد (المبلغ، العملة، التاريخ، ملاحظات)
- إضافة سداد جديد (يتطلب صلاحية `edit` على `travel_agency_suppliers`)
- حذف سداد

وتظهر أعلى نفس الصفحة خلاصة حساب هذا المورد تحديداً (مستحق/مدفوع/رصيد لكل عملة) — بذلك يمكن متابعة مورد واحد بالتفصيل، أو كل الموردين معاً من الصفحة العامة.

## 9. بوابة العميل (Client Portal)

يظهر تبويب **"My Bookings"** في بوابة العميل (بأيقونة طائرة) بمجرد تسجيل دخول العميل، ويحتوي ثلاث شاشات:

### أ. قائمة حجوزاتي
جدول بكل حجوزات العميل: اسم الباقة، الوجهة، عدد المسافرين، تاريخ السفر، وحالة الحجز (بشارة ملونة)، مع رابط "عرض التفاصيل" لكل حجز.

### ب. تصفح الباقات المتاحة والتقديم الذاتي (Self-Service)

هذه ميزة تفاعلية وليست عرضاً فقط: يمكن للعميل تصفّح **الباقات النشطة التي ما زال بها مقاعد شاغرة** (تُستبعد تلقائياً الباقات الممتلئة أو غير النشطة)، ثم فتح باقة معينة وتقديم طلب حجز ذاتي مباشرة من بوابته الخاصة، بإدخال عدد المسافرين وملاحظات اختيارية:

- إذا كانت الباقة لها تاريخ بداية محدد، يُستخدم هذا التاريخ تلقائياً كتاريخ السفر.
- يخضع الطلب لنفس تحقق سعة المقاعد المطبَّق على حجوزات لوحة التحكم — إن لم تعد هناك مقاعد كافية عند لحظة التقديم، يُرفض الطلب برسالة واضحة بدلاً من إنشاء حجز يتجاوز السعة.
- عند نجاح التقديم، يُنشأ حجز جديد بحالة **"قيد الانتظار"** باسم العميل نفسه، ويُحوَّل العميل مباشرة لصفحة تفاصيل هذا الحجز.
- هذا المسار **لا** يُنشئ فاتورة تلقائياً (خيار إنشاء الفاتورة التلقائي متاح فقط من نموذج الحجز في لوحة تحكم الموظفين) — يبقى الحجز "قيد الانتظار" لحين مراجعته وتأكيده من الموظف.

### ج. تفاصيل الحجز
عند فتح حجز معين (مع تحقق صارم من ملكية العميل له — أي عميل آخر لا يمكنه الوصول لحجز ليس له):

- الوجهة، عدد المسافرين، تاريخ السفر
- الإجمالي، معروضاً بعملة الباقة نفسها
- **رابط مباشر لعرض الفاتورة** المرتبطة بالحجز (إن وُجدت)
- ملاحظات الحجز

العميل **لا يرى** أي شيء من بيانات التفويج (خط السير، النقل، الجوازات، التأشيرات، الصور) — هذه كلها تبقى حصراً في لوحة تحكم الموظفين.

## 10. الترجمة (Localization)

الموديول مُترجم بالكامل للعربية والإنجليزية (ملفات لغة متوازية 1:1)، وتغطي جميع النصوص: الباقات، أنواع الباقات، الحجوزات (بما فيها نصوص إضافة العميل السريع)، التفويج (خط السير، النقل، الأعضاء، حقول الجواز والتأشيرة)، الموردين، حسابات وسدادات الموردين، وبوابة العميل.

## 11. ملخص قاعدة البيانات (الجداول)

| الجدول | الغرض |
|---|---|
| `travel_package_types` | أنواع الرحلات (مع بيانات افتراضية مزروعة عند التثبيت) |
| `travel_suppliers` | بيانات الموردين |
| `travel_packages` | الباقات السياحية (تشمل price وcost والعملة والمدة والمقاعد) |
| `travel_bookings` | حجوزات العملاء (تشمل الربط بالفاتورة وعدد المسافرين والإجمالي) |
| `travel_groups` | مجموعات السفر (التفويج) |
| `travel_group_itinerary_stops` | محطات خط السير (مدينة/أيام/فندق) لكل مجموعة — متعددة |
| `travel_group_transport` | أرجل النقل (ناقل/نوع/مرجع) لكل مجموعة — متعددة |
| `travel_group_members` | المسافرون ضمن كل مجموعة، مع بيانات الجواز والتأشيرة والصور |
| `travel_supplier_payments` | سدادات الموردين (المبلغ، العملة، التاريخ، ملاحظات) |

> جدولا `travel_group_itinerary_stops` و`travel_group_transport` حلّا محل حقول الفندق/الناقل الفردية التي كانت موجودة سابقاً مباشرة في جدول `travel_groups`، لدعم رحلات متعددة المدن ومتعددة وسائل النقل.

## 12. قيود وحماية البيانات (لمنع الحذف الخاطئ)

النظام يرفض تلقائياً حذف أي سجل ما زال مرتبطاً ببيانات أخرى، لحماية سلامة السجلات التاريخية والمالية:

- لا يمكن حذف **نوع باقة** مستخدم في أي باقة.
- لا يمكن حذف **باقة** مرتبط بها أي حجز.
- لا يمكن حذف **مورد** مرتبطة به أي باقة.
